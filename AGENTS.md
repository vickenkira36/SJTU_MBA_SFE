## Markdown 转 Word 标准命令

每次生成 docx 时**必须**先 pandoc 再 post-process，两步都不能省：

```bash
pandoc docs/chapterXX.md -o docs/chapterXX.docx \
  --reference-doc=docs/antai-template.docx \
  --lua-filter=docs/superscript-cite.lua \
  --resource-path=docs \
  -f markdown -t docx

python3 docs/post_process_docx.py docs/chapterXX.docx
```

- `--reference-doc`：安泰格式模板，定义所有样式
- `--lua-filter`：引用上标处理（正文中 `[N]` → 上标，参考文献列表中保持原样）
- `--resource-path=docs`：**必须**，否则 markdown 中的图片路径（如 `figures/fig1-1.png`）无法解析
- `post_process_docx.py`：**必须**，对所有表格设置 AutoFit + cantSplit + tblHeader，否则会出现"窄列文字一字一断"和"单元格内容跨页拦腰切断"的渲染问题

## 数学公式编号规则

正文公式编号**禁止**使用 `\tag{X-Y}`——这是 amsmath 扩展，pandoc 转 docx 时不支持，会导致整个公式 fallback 成 LaTeX 源码裸文本。改用 `\qquad\text{(X-Y)}`：

```markdown
$$d(i,j) = 2R \cdot \arcsin\sqrt{...} \qquad\text{(4-1)}$$
```

`\qquad` 是大空格，`\text{}` 把括号内容当文本处理。这种写法 pandoc 能稳定渲染，docx 中显示效果与传统 LaTeX 公式编号一致。

## 图片 DPI 规则

生成图片（Graphviz/matplotlib）后，**必须**设置 DPI 使图片在 Word 中宽度约 15cm（A4 文本区域宽度约 16cm）。pandoc 按图片内嵌 DPI 计算物理尺寸，DPI 不足会导致图片超出页面。

```python
from PIL import Image
TARGET_WIDTH_IN = 5.9  # ~15cm
im = Image.open(path)
needed_dpi = max(150, int(im.size[0] / TARGET_WIDTH_IN))
im.save(path, dpi=(needed_dpi, needed_dpi))
```

- Graphviz 生成时可用 `-Gdpi=150 -Gsize="6,10\!"` 控制输出尺寸
- matplotlib 生成时用 `fig.savefig(path, dpi=300, bbox_inches='tight')`
- 生成后务必用上述脚本校验并修正 DPI

## Markdown 表格格式规则

**优先使用 pipe table 格式**（`| | |` 加 `|---|---|`）。

```markdown
| 类别 | 符号 | 含义 |
|------|------|------|
| 参数 | K    | 辖区数量 |
```

**为什么不用 grid table**：grid table（`+--+--+` 边框形式）要求每行的 `|` 出现在严格对齐的字符位置上。但 pandoc 对中文字符按 East Asian Width 计算视觉宽度（中文 = 2），与 markdown 源文件的纯字符数对齐方式冲突——当中文密集列的内容跨越多行时，pandoc 会把列结构识别错乱（多算一列、内容串行），生成的 docx 表格完全错位。**这种错乱不会触发 pandoc warning**，因此除非肉眼检查 docx 否则不会发现。

历史教训：原本第二章表 2-1 和第四章表 4-1/4-2/4-3 都用了 grid table，docx 渲染时全部出现"表头被合并、数据行串到下一行"等问题，统一改 pipe table 后正常。

**关于 pipe table 列宽**：pandoc 转 docx 时会按内容估算列宽，中文场景下通常表现良好。如果需要精确控制列宽，在表前加属性 `: {col-widths="2,3,5"}` 或拆分长列。但绝大多数论文表格用默认估算就够了。

## 算法技术细节

算法的完整实现在 `src/lib/optimizer.ts`（约2150行 TypeScript），算法设计文档在 `docs/algorithm.md`。写作时必须参考这两个文件以确保技术描述的准确性。

### 算法流水线

```
原始医院数据 → 预处理(大医院虚拟拆分) → 按省份分组 → 每省独立执行:
  六层地理聚类(初始分配) → 模拟退火SA(均衡优化) → Hungarian匹配（全模式生效）
→ 汇总全国结果
```

### 关键算法模块

1. **预处理 — 大医院虚拟拆分**：index > 理想值×1.5 的医院等分为 n 个虚拟医院
2. **六层地理聚类**：Layer1 大医院独占+拆分分散 → Layer2 区县聚合 → Layer3 城市聚合（含一对一城市强制独立）→ Layer4 剩余城市收集 → Layer5 亲和图预合并 → Layer6 落单城市归属
3. **Maximin 种子选择**：score = minDist × (index / maxIndex)，兼顾地理分散和 index 权重
4. **模拟退火（SA）**：Move(60%) / Swap(40%)，邻接辖区间操作，100,000次迭代，纯均衡优化（不含历史惩罚）
5. **代价函数**：基于阈值的统一惩罚框架，(违反量 / threshold) × BASE_PENALTY
6. **Hungarian 匹配**：三层权重矩阵（医院级 index²/500、区县级 +100、城市级 +50），将匿名簇映射到历史辖区 ID
7. **约束体系**：硬约束（锁定、拆分分散、城市上限、空辖区）+ 软约束（Index均衡、容量、地理跨度、区县集中度）

### 方法论：两阶段法
本论文采用两阶段法作为唯一方法论，将均衡优化与历史延续性完全解耦：
- **第一阶段（均衡优化）**：六层地理聚类生成初始分配 → 模拟退火纯优化均衡性（SA 代价函数不含历史惩罚，500,000 次迭代，前 80% 标准退火 + 后 20% 贪心抛光）
- **第二阶段（历史匹配）**：用 Hungarian 算法将第一阶段产出的匿名簇映射到历史辖区 ID，最大化历史延续性

这种解耦设计的学术价值在于：均衡性不受历史数据干扰，历史延续性由独立的最优匹配保证，两个目标各自达到最优而非相互妥协。

## 学术写作课程指导规范

以下规范提取自安泰学院宋婷老师《学术写作、规范与伦理》课程四次教案（`docs/学术写作lecture01-04.pdf`），写作时必须严格遵循。

### Lecture 01：研究选题

1. **明确的 X 和 Y**：研究问题必须有明确的自变量（X）和因变量（Y），越具体越好。本论文中 X = 运筹优化算法（两阶段法），Y = 辖区分配的均衡性、稳定性与效率
2. **选题三要素**：有意义（理论+实践）、可行（数据可获取、方法可实现）、有创新（与现有研究的差异点）
3. **研究问题的层次**：从"是什么"→"为什么"→"怎么办"逐层深入
4. **避免选题过大**：不要试图解决所有问题，聚焦一个具体场景

### Lecture 02：文献综述

1. **文献综述不是"一本书"**：不要为凑字数硬塞与选题只有微弱关联的内容
2. **文献综述不是重复行业问题**：要总结文献里的研究结论，而非重复描述行业现象
3. **总-分-总结构**：主题句概括主旨 → 有机整合前人研究（层层深入、有逻辑，不要通篇罗列"谁谁谁研究了啥"）→ 批判性评价，总结不足，凸显本研究意义
4. **组织方法**：主题法（最常用，按中心思想划分 2-3 个类别）、时间顺序法、方法论法
5. **文献数量**：至少精读 20 篇与研究问题特别相关的"可信"论文
6. **文献来源**：优先 UTD 24、A- 期刊（英文）；《管理世界》《管理科学学报》等核心期刊（中文）
7. **引用规范**：即使重新解释了一段话，原始想法仍需引用；有疑问时，引用（宁多勿少）；确定一种引用格式，全文保持一致
8. **两种布局方式**：方式一（本论文采用）：第二章详写，第一章 1.3 高度概括；方式二：第一章着重写，第二章写历史发展、理论概述、应用案例

### Lecture 03：研究方法

1. **定性+定量结合**：建议案例/战略框架（定性）+ 数据分析（定量）
2. **方法只是工具**：选对合适且容易上手的方法，不要为炫技而选复杂方法
3. **警惕数据陷阱**：伯克森悖论（选择性偏差）、辛普森悖论（遗漏变量）、区分相关关系与因果关系
4. **对数据本身持怀疑态度**：数据既能澄清现实，也能混淆是非

### Lecture 04：论文分析与要求

1. **摘要**：500 字左右（中文），300 词左右（英文），3-6 个关键词；明确说明研究问题、理论依据、方法、内容及主要结论
2. **正文不少于 2.5 万字**（学校最低要求，本论文目标 4 万字）
3. **绪论至少占正文 10%**；常见错误：绪论过短、重要文献缺失、文献引用只是简单罗列无逻辑、否定前人工作
4. **绪论结构参考**：1.1 研究背景 → 1.2 研究的目的与意义 → 1.3 研究方法和框架 → 1.4 创新及不足 → 本章小结
5. **方法和过程说明**：用各种图表清晰描述变化和对比
6. **内容和分析是核心**：重要的不是发现问题，而是发现问题后如何改进；敏感性分析是加分项
7. **结论**：创新性（选题+结论普适性+弥补现有不足）+ 不足（数据局限、模型简化，要诚恳）+ 展望
8. **呈现形式**：多用图表，避免大段纯文字；参考行研报告获取数据支撑
9. **查重**：知网（https://check.cnki.net/），声明/目录/参考文献自动排除，致谢参与检测，语义级别检测
10. **写作优先级**：如果必须在"做完"和"做好"中二择一，选择"做完"；先完成全文框架，再逐章打磨

## 写作规范

### 格式要求

1. **封面格式**：中文题目 + 英文题目，学校/学院/专业/作者/导师/学号/班级/答辩日期
2. **摘要**：中文摘要 + 关键词，英文 ABSTRACT + KEY WORDS，各独立一页
3. **目录**：列至三级标题
4. **章节编号**：
   - 一级标题：`第一章`、`第二章`...（或 `第1章`、`第2章`...）
   - 二级标题：`1.1`、`1.2`...
   - 三级标题：`1.1.1`、`1.1.2`...
5. **每章结尾**：加 `X.Y 本章小结` 段落，概括本章要点并过渡到下一章
6. **图表编号**：`图 X-Y`、`表 X-Y`（X 为章号，Y 为序号），图题在下方，表题在上方
7. **公式编号**：右对齐，格式 `(X-Y)`
8. **参考文献**：统一列于论文末尾，正文中用 `[序号]` 标注，中英文文献混排
9. **页眉**：左侧 `上海交通大学 MBA 学位论文`，右侧 `论文简称`
10. **总篇幅**：正文约 4 万字（约 80-100 页）

### 行文风格规范

1. **学术性与实践性并重**：MBA 论文不是纯理论研究，必须紧密结合行业实践。每个理论观点都应有对应的业务场景说明
2. **语言风格**：
   - 使用第三人称或无主语句式（"本文提出..."、"研究发现..."），避免"我"
   - 语气客观严谨，避免口语化表达
   - 段落之间有逻辑过渡，使用"首先...其次...最后..."、"一方面...另一方面..."等连接词
   - 关键术语首次出现时给出中英文对照，如"销售力有效性（Sales Force Effectiveness, SFE）"
3. **数据呈现**：
   - 定量结论必须有数据支撑，使用表格和图表辅助说明
   - 对比分析使用"As-Is（现状）vs To-Be（优化后）"框架
   - 百分比保留一位小数，绝对值根据量级选择合适精度
4. **引用规范**：
   - 正文中引用格式：`Zoltners 和 Sinha (1983) 提出...` 或 `...已有研究证实[3]`
   - 中文文献引用：`张三(2020)指出...`
   - 禁止使用"文献[N]可知"、"由文献[N]得出"等写法，影响行文流畅性；统一用句末上标`[N]`
   - 即使重新解释了一段话，原始想法仍需引用；有疑问时，引用
   - 每章文献综述部分引用密度应较高，算法设计和实证章节引用密度可适当降低
   - 至少精读 20 篇与研究问题特别相关的可信论文
5. **算法描述规范**：
   - 先用自然语言描述算法思想和动机
   - 再给出数学符号定义和公式推导
   - 最后提供伪代码或流程图
   - 复杂公式需逐项解释各符号含义
6. **企业脱敏**：涉及具体企业数据时使用"某跨国药企"、"某省份"等脱敏表述
7. **查重意识**：避免大段引用原文，用自己的语言重新表述并标注引用来源
8. **降AIGC写作风格**（基于外部工具处理后的第一章提炼，所有章节必须遵循）：
   - **句式**：多用长句逗号连接（40-60字长句与10-20字短句交替），避免AI典型的"一句一义"短句节奏
   - **连接词**：避免连续"首先...其次...再次...最后..."排比，改用混合过渡词（"也就是说"、"于是"、"进而"、"不过"等）
   - **同义词替换**：优化→改良/改进、构建→创建/搭建、框架→体系/架构、显著→明显/突出、基于→依据/凭借/按照
   - **避免AI高频词**：不用"赋能"、"助力"、"深度融合"、"全方位"、"多维度"
   - **段落长度不均**：3-4句与7-8句段落交替，避免每段5句的均匀结构
   - **减少加粗标记**：只在关键术语处使用，不要每个要点都加粗
   - **数据引用后加解释从句**：引用数据后加解释性从句说明含义，不要直接句号结束
   - **迂回论证**：先铺垫背景再引出结论，用因果链条替代直接断言
   - **主语多样化**：避免"本研究"开头过多，用"此项研究"、"本文"、或省略主语替换

### 各章写作要点

| 章节 | 核心任务 | 注意事项 |
|------|---------|---------|
| 第一章 绪论 | 建立研究动机，"讲故事"，逻辑最重要 | 从宏观→微观层层递进；至少占 10%（4,500字）；含研究现状高度概括；列出创新点与不足 |
| 第二章 文献综述 | 总-分-总结构，找到研究缺口 | 不是"一本书"，不要硬塞内容；按主题法组织；最后批判性评价，凸显本研究意义 |
| 第三章 现状诊断 | 用案例和数据说明痛点的真实性 | 多用图表清晰描述；结合作者行业经验 |
| 第四章 模型构建 | 论文技术核心，两阶段法数学模型+算法设计 | 必须与 `optimizer.ts` 代码实现严格对应；公式推导完整；重点论证两阶段解耦的设计动机 |
| 第五章 实证分析 | 用数据证明两阶段法有效 | As-Is vs To-Be 对照；敏感性分析是加分项；重要的不是发现问题，而是如何改进 |
| 第六章 商业化 | 探讨落地路径和商业价值 | 人机协同机制、变革管理、SaaS 商业模式 |
| 第七章 结论 | 总结贡献，承认不足 | 创新性：选题+结论普适性+弥补现有不足；不足要诚恳；展望要具体可行 |

## 工作流程

当用户要求撰写论文内容时，按以下流程执行：

### 1. 确认范围
- 确认要写的章节编号和标题
- 确认该章节在论文整体中的定位和与前后章节的衔接关系

### 2. 收集素材
- 阅读开题报告中对应章节的描述（`docs/论文开题报告-陈一-124120935584.pdf`）
- 阅读相关参考文献（`docs/` 目录下的 PDF 文件，使用 `pdftotext` 提取）
- 阅读算法代码（`src/lib/optimizer.ts`）和算法文档（`docs/algorithm.md`）
- 参考论文案例的对应章节结构和行文风格

### 3. 撰写内容
- 严格按照论文大纲的章节结构撰写
- 遵循上述写作规范
- 数学公式使用 LaTeX 语法
- 每节末尾自然过渡到下一节

### 4. 输出格式
- 以 Markdown 格式输出，便于后续转换为 Word
- 图表位置用 `[图 X-Y: 图题]` 占位标注
- 公式使用 `$$..$$` 块级公式或 `$...$` 行内公式

### 5. 质量检查
- 检查与开题报告的一致性
- 检查算法描述与代码实现的一致性
- 检查引用文献的准确性
- 检查前后章节的逻辑衔接

## 数学符号约定

在论文中统一使用以下符号体系：

| 符号 | 含义 |
|------|------|
| $H = \{h_1, h_2, ..., h_N\}$ | 医院集合，共 N 家 |
| $T = \{t_1, t_2, ..., t_K\}$ | 辖区集合，共 K 个 |
| $I_i$ | 医院 $h_i$ 的综合价值指数（Index） |
| $I^*$ | 理想辖区 Index 值（indexTarget） |
| $[I_{min}, I_{max}]$ | 辖区 Index 允许范围 |
| $x_{ij}$ | 分配变量，医院 $h_i$ 分配给辖区 $t_j$ 的比例（0或1，拆分时为 $1/n$） |
| $S_j = \sum_i x_{ij} \cdot I_i$ | 辖区 $t_j$ 的 Index 总值 |
| $d(h_i, h_k)$ | 医院 $h_i$ 与 $h_k$ 之间的 Haversine 距离（km） |
| $C_{max}$ | 单辖区医院数量上限 |
| $M_{max}$ | 单辖区城市数量上限 |
| $\mathcal{A}(j)$ | 辖区 $t_j$ 的邻接辖区集合 |
| $\mathcal{L}(i)$ | 锁定医院 $h_i$ 的允许辖区集合 |
| $T_0, \alpha$ | 模拟退火初始温度和冷却系数 |
| $w_{cj}$ | Hungarian 匹配权重矩阵中簇 $c$ 到历史辖区 $j$ 的权重 |

## 禁止事项

1. **不得编造数据**：实证分析中的数据必须来自用户提供的真实脱敏数据或代码运行结果
2. **不得偏离开题报告**：论文结构和研究内容必须与已通过的开题报告保持一致
3. **不得虚构引用**：所有引用必须来自 `docs/` 目录下的真实文献或用户补充的文献
4. **不得过度技术化**：MBA 论文的读者是管理学背景，算法描述需兼顾可读性
5. **不得忽略商业视角**：每个技术决策都应解释其背后的业务动机
